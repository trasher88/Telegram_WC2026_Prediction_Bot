# Project Help — структура проекта

Краткая карта Telegram-бота турнира прогнозов ЧМ-2026.

Проект разделён на слои:

- `handlers/` — Telegram-команды, callback-кнопки и пользовательские сценарии;
- `repositories/` — SQL-запросы и доступ к SQLite;
- `services/` — бизнес-логика: прогнозы, подсчёт очков, рассылки, статистика;
- `utils/` — форматирование, тексты, флаги, русские названия команд;
- `scheduler.py` — фоновые задачи: синхронизация API, блокировка прогнозов, дневные уведомления;
- `web_admin.py` — минимальный read-only web admin через FastAPI.

---

## Точка входа и конфигурация

| Файл | За что отвечает |
|---|---|
| `main.py` | Главная точка запуска. Создаёт `Bot` и `Dispatcher`, подключает router’ы, инициализирует БД, при необходимости запускает синхронизацию матчей, scheduler и polling Telegram. Также регистрирует команды в меню Telegram. |
| `config.py` | Централизованная конфигурация. Читает переменные окружения: `BOT_TOKEN`, `FOOTBALL_DATA_API_TOKEN`, `ADMIN_IDS`, `APP_TIMEZONE`, `WEB_ADMIN_API_KEY`, `DB_PATH`, `ENABLE_API_SYNC`. |
| `.env.example` | Пример переменных окружения. Сам проект не загружает `.env` автоматически, если в код не добавлен `python-dotenv`. Переменные нужно передавать через IDE, терминал, systemd, Docker или вручную подключить загрузку `.env`. |
| `.gitignore` | Исключает секреты, SQLite-базы, виртуальное окружение, кэш Python и файлы IDE из git/архивов. |
| `requirements.txt` | Python-зависимости проекта: `aiogram`, `aiosqlite`, `httpx`, `apscheduler`, `fastapi`, `uvicorn` и др. |

## Работа с внешним API

| Файл | За что отвечает |
|---|---|
| `api_client.py` | Клиент football-data.org. Получает матчи ЧМ через football-data API с токеном `FOOTBALL_DATA_API_TOKEN`. |
| `scheduler.py` | Использует `APIClient` для синхронизации матчей, статусов и счёта. |
| `test_api.py` | Dev-скрипт для ручной проверки доступа к football-data API. |

Важно: `ENABLE_API_SYNC=0` отключает автоматическую синхронизацию при запуске и периодический sync. Команда `/force_sync` должна быть защищена от запуска при `ENABLE_API_SYNC=0`, чтобы тестовая база не перезаписывалась API.

## Scheduler и фоновые задачи

| Файл | За что отвечает |
|---|---|
| `scheduler.py` | Синхронизирует матчи с API, обновляет `matches`, планирует lock прогнозов за 5 минут до матча, запускает обработку завершённых матчей, рассылает результаты и планирует дневные уведомления. |

Фоновые задачи:

| Задача | Логика |
|---|---|
| API sync | Каждые 5 минут, если `ENABLE_API_SYNC=1`. |
| Prediction lock | За 5 минут до начала каждого матча. |
| Daily digest | Каждый игровой день в 13:00 МСК всем игрокам. |
| Missing predictions reminder | В 18:00 МСК только тем, у кого не все прогнозы поставлены. |
| Admin daily report | В 18:30 МСК администраторам. |

Игровой день считается по Москве: с 13:00 текущего дня до 12:59 следующего дня.

## Правило счёта в плей-офф и пенальти

Для матчей на вылет действует турнирное правило:

- если матч завершился без серии пенальти, используется обычный итоговый счёт;
- если была серия пенальти, победителю серии добавляется `+1` гол;
- пример: после игры 1:1, первая команда выиграла пенальти — для прогнозов считается 2:1;
- пример: после игры 1:1, вторая команда выиграла пенальти — для прогнозов считается 1:2.

Эта логика применяется при синхронизации API до записи `home_score` / `away_score` в таблицу `matches`.

## Telegram handlers

| Файл | За что отвечает |
|---|---|
| `handlers/user.py` | Агрегатор пользовательских router’ов. Подключает `common`, `matches`, `predictions`, `my_predictions`, `admin`. |
| `handlers/common.py` | Общие команды: `/start`, `/rename`, `/help`. Регистрирует пользователя, запрашивает турнирное имя при первом запуске, позволяет переименоваться. |
| `handlers/matches.py` | Команда `/matches`, выбор стадий и туров, вывод расписания с флагами, русскими названиями и московским временем. |
| `handlers/predictions.py` | Команда `/predict`, выбор стадии, выбор матча, быстрые кнопки счёта, ручной ввод прогноза, сохранение прогноза. |
| `handlers/my_predictions.py` | Команда `/my_predictions`. Показывает прогнозы пользователя, будущие матчи, завершённые матчи и начисленные очки. |
| `handlers/leaderboard.py` | Команда `/leaderboard`. Показывает очки, точные счета, количество прогнозов и позицию пользователя. При равенстве очков выше игрок с большим числом точных счетов. |
| `handlers/player_stats.py` | Пользовательские команды `/form` и `/my_stats`. Показывает форму за последние 10 завершённых прогнозов и общую карточку игрока. |
| `handlers/admin.py` | Админ-команды: `/admin`, `/force_sync`, `/set_score`, `/rebuild_scores`, `/match_info`, `/broadcast`, `/users`, `/stats`, `/match_ids`, `/test_lock`. Доступ ограничен через `ADMIN_IDS`. |
| `handlers/keyboards.py` | Общие inline-клавиатуры: выбор стадий, групповых туров и быстрый выбор счёта прогноза. |

## Repository-слой

| Файл | За что отвечает |
|---|---|
| `repositories/users.py` | Регистрация пользователя, хранение `display_name`, обновление Telegram-данных, получение пользователей для рассылок. |
| `repositories/matches.py` | Матчи по стадиям/турам, доступные матчи для прогноза, выставление счёта, получение match info, список ID матчей, lock матча. В прогнозы не должны попадать матчи с неизвестными командами. |
| `repositories/predictions.py` | Прогнозы пользователя, ID уже спрогнозированных матчей, количество прогнозов на матч. |
| `repositories/leaderboard.py` | Данные leaderboard: очки, количество прогнозов, количество точных счетов и ranking. |
| `repositories/player_stats.py` | Данные для `/form` и `/my_stats`: последние завершённые прогнозы, все завершённые прогнозы, открытые прогнозы. |
| `repositories/admin_stats.py` | Данные для админ-статистики: пользователи, активность, матчи, прогнозы, обработанные матчи. |

## Services — бизнес-логика

| Файл | За что отвечает |
|---|---|
| `services/predictions.py` | Проверка доступности прогноза, запрет прогноза на неизвестные команды, проверка lock/status/deadline, атомарное сохранение прогноза, расчёт результата прогноза. |
| `services/scoring.py` | Правила начисления очков, обработка завершённого матча, защита от повторной обработки, полный пересчёт очков. |
| `services/match_broadcasts.py` | Рассылки по матчу: закрытие прогнозов, результаты прогнозов после завершения матча. Следит за повторной отправкой через `match_notifications`. |
| `services/daily_notifications.py` | Дневные уведомления: список матчей дня, напоминания тем, кто не поставил прогнозы, админ-отчёт по поставившим и не поставившим прогнозы. |

## Правило навигации по проекту

1. Команда или callback от Telegram приходит в `handlers/`.
2. Handler валидирует входные данные и вызывает `repositories/` или `services/`.
3. `repositories/` читают/пишут SQLite.
4. `services/` реализуют бизнес-правила.
5. `utils/` форматируют данные и хранят общие словари/тексты.
6. `scheduler.py` работает отдельно от пользовательских команд и обслуживает фоновые процессы турнира.
